## CRITICAL

### Unauthenticated account takeover: /api/auth/google/link trusts a forgeable plaintext cookie and links any entrant
- **Location:** `app/api/auth/google/link/route.ts:16`  ·  **Dimension:** security  ·  **Found by:** ?
- **What:** The link endpoint's only gate is the presence of the `google_pending_email` cookie, which is a plain, unsigned value the client controls (set in app/api/auth/google/callback/route.ts:101 with no signature/HMAC). An attacker can simply send `Cookie: google_pending_email=anything@example.com` and POST any `entrant_id` — no Google OAuth needed at all. The endpoint then overwrites that entrant's `google_email` (lib/draftAuth.ts:171-178 has no `google_email IS NULL` guard) and mints a full 14-day session as that entrant (line 37-46), including admin entrants (`is_admin` is honored everywhere downstream). Valid entrant_ids are freely enumerable because `/api/season/2026/leaderboard` (app/api/season/[year]/leaderboard/route.ts) performs no auth check and returns every member's `entrant_id`. Since API-route auth checks are the app's stated security boundary (service-role client bypasses RLS), this is a complete break of it. Even without a malicious actor, any Google user in the world can complete the real OAuth flow and claim any member's identity via the picker.
- **Evidence:** const pendingEmail = cookieStore.get(GOOGLE_PENDING_EMAIL_COOKIE)?.value; ... const entrant = await getEntrantById(entrantId); if (!entrant) {...} await linkGoogleAccount(entrantId, pendingEmail); const session = await createDraftSession(entrant.pool_id, entrantId);  // no check that the entrant is unlinked, no signature on the cookie, no allowlist of who may link
- **Suggested fix:** Sign the pending-email cookie (HMAC with a server secret, or store a random token server-side mapped to the verified email). In the link handler, reject if the target entrant already has a google_email set, and restrict linking to emails on a pre-seeded allowlist (e.g. only entrants whose expected email matches, or require admin approval). Also require auth on the leaderboard route or stop exposing entrant_id there.

### Draft page never re-polls /api/draft-state, so 'on the clock' status, countdown, and pick buttons go permanently stale
- **Location:** `app/draft/page.tsx:181`  ·  **Dimension:** data-flow  ·  **Found by:** ?
- **What:** The effect that fetches /api/tournament-meta and /api/draft-state (lines 150-181) has deps [poolId, selectedTournament] only — it is NOT re-run on the 30s refreshTick. The 30s auto-refresh (lines 183-274) reloads entrants, session, golfers, and picks, but never draftState or draftOpen. draftState is otherwise only updated inside reloadPicks(), which runs after the user's OWN pick. So a phone user waiting for their turn never sees the header flip to "You're on the clock", isOnClock stays false, and every Draft button stays disabled showing "Wait" until they hard-reload the page. Compounding it: useAutoRefreshValue(30000, draftOpen) (line 111) is gated on draftOpen, which is false on initial load if the commissioner hasn't opened the draft yet — so users who open the page early get zero polling at all and never see the draft start. This also matters server-side: draft progression (timer expiry, auto-picks) only happens inside advanceDraftState, which is only invoked by GET /api/draft-state — with no client polling it and no cron (verified: no vercel.json/app/api/cron), an expired turn sits unprocessed until someone loads a page fresh.
- **Evidence:** useEffect(() => { ... loadDraftState() ... }, [poolId, selectedTournament]);  // line 181 — refreshTick absent. vs. picks effect: }, [poolId, entrantNames, refreshTick]);  // line 274. And line 111: const refreshTick = useAutoRefreshValue(30000, draftOpen);
- **Suggested fix:** Add refreshTick to the draft-state effect deps (or move the two fetches into the same polling effect as picks), and make the polling hook always enabled (or enabled at a slower rate when draftOpen is false) so clients notice the draft opening and turn changes.

- **What:** On first Google sign-in the "Who are you?" dropdown defaults to `rows[0]` — the season leaderboard is sorted by total_points, so the default selection is the current points leader, not the person signing in. ### Check-then-insert race in /api/draft-picks/add lets an on-clock entrant record two picks in one turn
- **Location:** `app/api/draft-picks/add/route.ts:43`  ·  **Dimension:** functionality  ·  **Found by:** ?
- **What:** Turn validation (line 43, via advanceDraftState at line 36) and the insert (line 86) are separate non-transactional Supabase round-trips spanning ~5 sequential queries. Two near-concurrent requests from the same entrant (double-tap on a phone before React re-renders the disabled state, or phone+laptop) both pass the current_entrant_id check. If request B reads existingPicks (line 66) after request A's insert commits, B computes nextPickNumber = A's+1 and its insert succeeds — the DB unique constraints (pool_id, golfer) and (pool_id, entrant_name, pick_number) in supabase/golfers_schema.sql lines 24-25 do not block this interleaving. The entrant consumes two global pick slots, so on the next advanceDraftState call currentPick jumps past the next entrant's snake slot and they silently lose their turn. The client-side guard (savingPicks) only disables buttons after a re-render, so a fast double-tap is realistic on mobile.
- **Evidence:** if (draftState.current_entrant_id !== session.entrant.entrant_id) { ... 409 }  // line 43 ... const nextPickNumber = (existingPicks?.length ?? 0) + 1;  // line 84 ... await supabaseAdmin.from("draft_picks").insert({ ... pick_number: nextPickNumber });  // line 86 — no transaction or conditional insert tying the pick to the turn slot.
- **Suggested fix:** Serialize picks per pool, e.g. a Postgres function (RPC) that re-validates the on-clock entrant and inserts atomically with an advisory lock on pool_id, or add a unique (pool_id, overall_pick_number) column derived from the turn being consumed.

### advanceDraftState performs unserialized writes from a public GET — concurrent calls double-insert or 500 during auto-draft turns
- **Location:** `lib/draftOrder.ts:477`  ·  **Dimension:** functionality  ·  **Found by:** ?
- **What:** GET /api/draft-state (app/api/draft-state/route.ts line 41) calls advanceDraftState, which inserts auto-picks with no locking. When an auto-draft entrant is on the clock and two clients load the draft page simultaneously, both executions compute the same getHighestAvailableGolfer and call insertAutoPick: the loser hits the unique (pool_id, golfer) constraint, insertAutoPick throws (line 280), and that user's GET returns 500 — on the draft page the catch (line 169-174) then sets draftOpen=false/draftState=null, showing the draft as "paused". Worse interleaving: if call B's getHighestAvailableGolfer runs after call A's insert commits, B picks the NEXT golfer with pick_number = count+1 and succeeds, giving the auto-draft entrant two picks for one turn slot and skipping the following entrant (same slot-consumption effect as the add-route race). Both paths mutate draft_state via blind upserts (line 510) so last-writer-wins on the turn pointer.
- **Evidence:** if (currentEntrant.auto_draft_enabled) { const golfer = await getHighestAvailableGolfer(poolId); if (golfer) { await insertAutoPick(poolId, currentEntrant, golfer); } currentPick += 1; ... continue; }  // lines 476-484, reachable concurrently from every GET /api/draft-state
- **Suggested fix:** Guard the auto-pick + pointer advance with a pg advisory lock or an RPC, or make insertAutoPick tolerate unique-violation errors (treat as already-picked and re-read state) instead of throwing.

### Score sync deletes all stored round scores before insert — a failed insert wipes the tournament's data
- **Location:** `app/api/slashgolf/leaderboard/sync/route.ts:90`  ·  **Dimension:** data-flow  ·  **Found by:** ?
- **What:** The sync route deletes every tournament_round_scores row for the pool/tournament (lines 90-98) and then inserts the fresh rows (lines 100-108) with no transaction. ### Solo tracker stores only HH:MM — entries older than 24h resurrect as 'recent' every day
- **Location:** `public/drink-tracker-solo.html:276`  ·  **Dimension:** data-flow  ·  **Found by:** ?
- **What:** Entries persist to localStorage with a bare 'HH:MM' string (add-drink at line 997, add-drug at line 1014). parseTime() rebuilds the Date on every render using TODAY's date, shifting to yesterday only if the time is in the future. So a drink logged Friday at 20:00 and viewed Saturday at 21:00 resolves to Saturday 20:00 = '1h ago': it re-enters BAC/caffeine math, drink counts, hydration, and the timeline as if just consumed. Entries never age out past ~24h; every evening the tracker shows phantom BAC from prior days until the user taps 'Reset all entries'. This is the storage-schema root defect of the standalone app.
- **Evidence:** function parseTime(str) {
  if (!str) return new Date();
  const [h, m] = str.split(':').map(Number);
  const d = new Date();
  d.setHours(h, m, 0, 0);
  // If time is in the future, assume yesterday
  if (d > new Date()) d.setDate(d.getDate() - 1);
  return d;
}
- **Suggested fix:** Store a full epoch timestamp on each entry (like the shared app's ISO occurred_at) and keep HH:MM only as display/input sugar; migrate the 'chris-tracker-v2' schema on load.

## MEDIUM

### Open redirect: OAuth callback redirects to attacker-controlled returnTo
- **Location:** `app/api/auth/google/callback/route.ts:85`  ·  **Dimension:** security  ·  **Found by:** ?
- **What:** `returnTo` originates from the query string of /api/auth/google (route.ts:18 `url.searchParams.get("returnTo") || "/"`), is round-tripped through the OAuth state parameter, and is then passed to `NextResponse.redirect(new URL(dest, url.origin))` with no validation. If `dest` is an absolute URL (`https://evil.com`) or protocol-relative (`//evil.com`), `new URL(dest, base)` ignores the base and the user is redirected off-site immediately after completing a legitimate Google sign-in — a classic phishing primitive (the same unvalidated redirect happens at line 98 for the /link-account path, and client-side in app/sign-in/page.tsx:26 via `router.replace(returnTo)`). For a private pool the blast radius is small, but the fix is one line.
- **Evidence:** const dest = entrant.welcomed_at ? returnTo : `/link-account?returnTo=${encodeURIComponent(returnTo)}`; const response = NextResponse.redirect(new URL(dest, url.origin));
- **Suggested fix:** Validate returnTo before use: accept only same-origin paths, e.g. `const safe = returnTo.startsWith("/") && !returnTo.startsWith("//") ? returnTo : "/";` at the top of the callback (and in the sign-in/link-account pages).
